## 0. 一句话目标
做一个**干净、克制、专业、有"可信交易感"的虚拟商品交易平台(发卡系统)**的完整 UI——四端统一设计语言,桌面与移动都打磨到位。**廉价感 = 不敢付款**,这是底线。

---

## 1. 产品与角色

**产品**:虚拟商品自动发货平台。商家在平台开店卖虚拟商品,买家下单付款后**自动发货**。商品分四类,发货方式各异(括号内是**对 UI 的影响**,设计师据此区分):
- ⚡ **卡密**:一卡一售——每张卡密被买走即失效、不再卖出。(UI:库存 = 未售卡数;商品卡按库存显示"现货/库存N/少量/缺货";结果页给**卡密列表**+逐条复制+复制全部;后台卡密列表**脱敏**显示)
- ☘️ **知识**:多章节图文内容,**购后立即解锁全部章节、在站内阅读**(不下载、无试读、无需审核)。(UI:详情页给**章节目录预览(仅标题)**;结果页给「前往阅读」入口,打开内置阅读器,左目录右正文)
- 💎 **资源**:文件下载,购后赠送**限时签名下载链(默认 30 分钟有效,过期可重新查单获取)**。(UI:结果页给「下载资源」按钮 + 有效期提示;过期态有重新获取入口)
- 👑 **权益**:权益码一权一售(如会员卡/序列号),机制同卡密但语义是"权益码"。(UI:后台该商品的"卡密"标签自动变「权益码」;结果页给权益码 + 复制)

> **术语表**(给不熟业务的设计师):**店招**=店铺首页顶部的视觉入口区(封面横幅+圆形logo+店名+认证徽章+三联统计)。**卡密**=一串可复制的密文/激活码。**查单/取卡**=买家凭订单号+邮箱(或查单密码)查询订单状态并领取已购内容。**一卡一售/一权一售**=同一张卡/权益码只卖给一个订单,售出即不可再卖。**佣金**=平台从每笔成交按费率抽成。**保证金**=商户在平台的押金,展示给买家做信任背书。

**四端 / 四类用户**:
1. **买家商城(C 端)**——游客买家浏览下单取货。**响应式,移动端必须好看**(多数买家用手机)。
2. **门户站**——平台官网:首页/资讯/常见问题/禁售目录/开通小店入口。
3. **商户后台(B 端)**——卖家管理自己的店。桌面为主,兼顾平板。
4. **平台后台(运营端)**——平台方管理全平台。桌面端、数据密集。

## 2. 设计理想 / 风格方向(最重要,请细读)

- **整体气质**:浅色、通透、留白充足;现代 SaaS 质感;**金融级信任感**,不花哨。
- **主色**:信任蓝(参考 `#2F6BFF` 一带,可微调);中性灰阶做骨架;成功绿/警告橙/危险红/信息蓝做语义。
- **卡片化**:白底卡片 + 柔和阴影 + 中等圆角(8–12px),清晰分区。
- **后台**:信息密度高但**不拥挤**——明确的层级、分组、留白与对齐;表格、数据卡是主力。
- **商城**:**商品是主角**,卡片要饱满(别大片空白),价格醒目,信任元素(已认证/担保/已售)清晰;移动端单列、桌面 2–4 列。
- **微交互克制但完整**:hover / 按下 / 加载骨架 / 空状态 / 错误态 / Toast 都要设计到。
- **一致性**:四端共用同一套 token 与组件,换皮也是同一血统。

**明确要避免**(发卡站通病,务必规避):
- 大面积花哨渐变、霓虹/廉价配色
- 间距随意、不在刻度上、元素不对齐
- 组件风格不统一(同一种按钮三种样子)
- 商品卡空洞(一个小图标飘在大白框里)
- 空状态/错误态难看或缺失
- 移动端塌掉、信息溢出、点击区过小

## 4. 技术约束(决定能否 1:1 落地,务必遵守)

前端是 **React 18 + Vite**,样式用 **CSS 变量(设计 token)+ 组件化(内联样式 / 设计系统)**。因此:
- **所有视觉属性用 token 表达并给具体数值**:颜色、字号、行高、字重、间距、圆角、阴影、层级 z-index。
- **间距走统一刻度**:建议 4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48。
- 已有基础组件,请基于这些做设计(并补全状态):
  `Button`(primary/secondary/ghost/neutral/danger × sm/md/lg)、`Input`、`Select`、`Textarea`、`Badge`、`Pill`、`Modal`、`DataTable`(表格)、`Panel`(带标题/副标题/操作区的卡)、`StatCard`(数据卡,含实底变体)、`Tabs`、`Toolbar`、空状态、`Toast`、进度条(CheckoutSteps)、客服浮窗。
- **响应式断点**:至少给 **移动(≤480)/ 平板(≤960)/ 桌面(≥1200)** 三档;C 端商城三档都要稿,后台至少桌面 + 移动抽屉态。
- 字体:中文优先系统无衬线(可指定 fallback 链);数字建议用 tabular 等宽对齐金额。

- 弹窗 Modal(标题/正文/底部按钮区);

- 空状态(带插画)、错误态(带重试)、Toast。

## 8. 关键流程与状态(别只画"正常态")
- **下单流程**:浏览 → 详情(试算优惠)→ 下单 → 支付(扫码/跳转)→ 发货中轮询 → 取货(四类型差异)。每步的加载/失败态。
- **订单状态机**:待支付 / 已支付发货中 / 已发货 / 已关闭过期 / 已退款 / 异常待人工——每态都有对应展示与文案。
- **售后**:发起投诉 → 商户回复 → 申请平台介入 → 平台裁决(可退款)。
- **空 / 错 / 加载**:每个列表与详情都要这三态。

## 附录 C · 全局状态与边界清单(每页都要画的"非正常态")
- **四态标配**:每个列表/详情都要 **加载骨架 / 空态(带插画+引导)/ 错误态(文案+重试)/ 内容态**。
- **表单校验**:输入框 默认/聚焦/报错(红边+下方红字)/禁用/只读;邮箱格式、必填、字数上限实时提示;**密码框带显隐眼睛**。
- **按钮**:loading(转圈+禁点)、disabled(置灰)、危险操作(退款/删除/拉黑/作废)走二次确认弹窗。
- **订单状态机展示**(6 态各有样式与文案):待支付(去支付)/已支付·发货中(轮询)/已发货(展示内容)/已关闭过期/已退款(原路退回)/异常待人工(醒目,绝不显示去支付)。
- **登录**:账号不存在/密码错/**账户被冻结**/系统错 各有文案;登录按钮 loading;**忘记密码**入口(本期可仅留入口或"联系平台")。**注册**:用户名/店铺标识格式与占用校验、邀请码无效提示、密码强度、协议勾选。
- **支付页**:二维码过期可刷新;轮询超时(如 5 分钟)提示;切支付方式金额同步。
- **商户**:**审核中**(只读、横幅提示)/**被冻结**(禁操作);商品**待审/违规下架**态;限时折扣价校验(须 <单价、起<止);知识类无章节禁发布;资源地址失效提示。
- **取货页**:查单凭证错误的失败态 + 重试;资源链过期重取;权益码已使用/过期态;复制失败兜底提示。
- **全局**:网络离线/500/超时 统一错误规范(轻提示用 **Toast 右上、3 秒自动消失、可堆叠**;整页失败用**居中错误态 + 重试**);弹窗点遮罩/ESC 关闭策略、有未保存改动时二次确认;移动端表格**横向滚动**提示、长文案行数限制+展开。
